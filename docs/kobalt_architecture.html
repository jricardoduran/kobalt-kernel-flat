Doctrina central
La verdad nace en local · La red es persistencia opaca y pasiva · El servidor desaparece después del login
Patrón Universal
aplica a storages · proxies · relays · models sin excepción
①
Registro
Admin configura credenciales. Se genera un service_key aleatorio. Todo en services/*.json — secreto, nunca sale del servidor.
→
②
Publicación
?action=active genera JSON público con metadata sin credenciales. Accesible sin autenticación por cualquier nodo.
→
③
Consumo
Nodo descarga JSON, guarda configs en IDB cifradas con D. Opera directo con cada servicio. El servidor puede caer — el nodo no lo nota.
Invariante fundamental: el nodo genera por cada llamada HMAC(service_key, id‖window, 16) — efímero, expira en ≤ 30 s. El service_key vive µs en el nodo (Tipo B). Capturado = inútil. Servidor puede caer.

Proxies
stateless
el nodo llama directo vía proxies · el servidor puede caer
El nodo siempre llama directo al endpoint registrado. El proxy es un tipo de servicio — su endpoint traduce autenticación (HMAC → auth nativa). Wasmer es efímero por diseño: stateless, sin estado que perder, reinicio instantáneo. Eso lo hace candidato perfecto para fallback: el servidor Kobalt puede caer, GitLab y R2 siguen respondiendo vía Wasmer.
⚡

runtime: php · efímero · stateless
El mismo proxy.php desplegado en Wasmer. Sin código diferente. El nodo lo intenta automáticamente via fallback_url si el servidor principal no responde. GitLab y R2 siguen vivos — son servicios independientes.

Invariantes del Sistema
aplican a las 4 categorías sin excepción
I1
Adición pura. Añadir un tipo nuevo en cualquier categoría no modifica ningún archivo existente. Solo se crea una carpeta con definition.json + register.php + endpoint.
I2
JSON público sin secretos. action=active nunca expone service_key, api_key ni ninguna credencial. Solo metadata operativa.
I3
Token efímero. HMAC(service_key, id‖window, 16) expira en ≤ 30 s. El service_key vive µs en el nodo (Tipo B). Capturado = inútil.
I4
Nodo autónomo post-login. El servidor Kobalt puede caer. El nodo tiene todo en IDB cifrado con D. Cada servicio es independiente de los demás.
I5
services_api.php es función pura. Solo lee filesystem. Nunca escribe. Lag de sincronización = cero matemático.
I6
El nodo es ciego al tipo. Solo envía (service_id, token, data). No sabe si detrás hay filesystem, GitLab, R2, Wasmer o Groq.
I7
Degradación graciosa. La caída de cualquier servicio no afecta a los demás. El nodo opera con los disponibles y degrada a local_only si todos caen.
I8
Mismo patrón, 4 categorías. storages · proxies · relays · models comparten estructura de registro, publicación y consumo.
